Pharma Trail — where the data lives & how each consumer reads it

Two databases, two jobs. ClickHouse is the complete analytical source of truth (columnar / OLAP) — it answers open-ended "scan & aggregate" questions. Neon Postgres is a trimmed, pre-computed read-model (row store / OLTP) — it answers two fast, narrow screens. Row counts & sizes below are live from each DB (2024 program year).

How the shared drug_key is built (the bridge between the two datasets)

Open Payments and Part D speak different vocabularies — OP records the brand product a rep paid for (drug1..5); Part D records the brand it was dispensed as (brnd_name) plus its generic (gnrc_name). We never match one dataset's name to the other's. Instead each raw name is normalized and looked up in drug_map to earn the same drug_key — and that key is the only thing the join uses.

Once both sides carry drug_key, the link is trivial: rx_by_npi_drug LEFT JOIN pay_by_npi_drug USING (drug_key, npi) → a doctor's prescribing lined up with the payments they got for that exact drug. The brand is the bridge; gnrc_name only does work for the Metformin control.

Why split this way? ClickHouse scans millions of rows per query cheaply → perfect for the MCP's arbitrary aggregations and the live /explore dashboard. Postgres is built for indexed point lookups → perfect for "find my doctor" + rendering one doctor's page. Neon holds a derived subset (no raw events), so the MCP can't answer the interesting questions from it — that's why the MCP reads ClickHouse only.
Pipeline: full CMS CSVs → scripts/ filter & roll-up → ClickHouse (all tables) and Neon (the 4 read-model tables). Data is read-only public CMS data, program year 2024.
